## Core Terminology

* **Query Plan** — The step-by-step execution strategy generated by the PostgreSQL planner before running a query. Each step describes how data is accessed and combined.
* **Sequential Scan (Seq Scan)** — A strategy where PostgreSQL reads every row in a table from start to finish. Efficient on small tables; expensive on large ones.
* **Index Scan** — A strategy where PostgreSQL uses an index structure to locate only the matching rows directly, avoiding a full-table read.
* **Index Only Scan** — An optimized Index Scan where all requested columns are available within the index itself, so the table is never touched.
* **`EXPLAIN`** — A command that displays the planned execution tree for a query without actually running it.
* **`EXPLAIN ANALYZE`** — Executes the query and displays both the planned estimates and the actual measured runtime for each step.
* **Selectivity** — A measure of how many rows a filter condition returns relative to the total. High selectivity (few matching rows) is where indexes provide the most benefit.
* **Covering Index** — An index that contains all the columns required by a query, enabling an Index Only Scan and eliminating the need to read the heap table.
* **Dead Tuples** — Row versions left behind after `UPDATE` or `DELETE` operations. They consume disk space and distort the planner's row-count estimates until cleaned up by `VACUUM`.

## Task 2: The Cost of SELECT *

### Objective
Demonstrate why fetching all columns wastes I/O and how targeted column selection combined with a covering index eliminates heap access entirely.

---

### Step 1: Measure the Wide Query

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT * FROM order_items WHERE product_id = 'prod_0100';
```

Record the total runtime and the number of `Buffers: shared hit` blocks read.

---

### Step 2: Narrow the Column List

Rewrite the query to select only `order_item_id`, `quantity`, and `item_price`:

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT order_item_id, quantity, item_price
FROM order_items
WHERE product_id = 'prod_0100';
```

---

### Step 3: Add a Covering Index

```sql
CREATE INDEX idx_order_items_covering
ON order_items(product_id)
INCLUDE (order_item_id, quantity, item_price);
```

Re-run the narrowed query from Step 2 after creating the index.

**Verification Requirement:** The execution plan must show `Index Only Scan` against `idx_order_items_covering`. The `Heap Fetches` counter in the output should be `0`, confirming the table itself was not accessed.

## Task 3: Functions in WHERE — The Index Killer

### Objective
Observe how wrapping a column inside a function in a `WHERE` clause silently disables index usage, and fix it using a range predicate.

---

### Step 1: Run the Problematic Query

```sql
EXPLAIN ANALYZE
SELECT order_id, order_date
FROM orders
WHERE EXTRACT(YEAR FROM order_date) = 2024;
```

Record the scan type and runtime.

---

### Step 2: Rewrite Using a Range Predicate

```sql
EXPLAIN ANALYZE
SELECT order_id, order_date
FROM orders
WHERE order_date >= '2024-01-01 00:00:00'
  AND order_date <  '2025-01-01 00:00:00';
```

| | EXTRACT() version | Range predicate version |
|---|---|---|
| Scan type | | |
| Actual time (ms) | | |

**Verification Requirement:** The range predicate version must show `Seq Scan` replaced by a partition-aware plan that scans only `orders_2024`, confirming both function-avoidance and partition pruning are active simultaneously.

> **Question:** Explain in your own words why `EXTRACT(YEAR FROM order_date) = 2024` prevents index usage while the range predicate does not. What property of the column must be preserved for an index to be applicable?

---

### Step 3 (Advanced): Functional Index

If filtering by extracted year is a genuine business requirement that cannot be rewritten, a functional index can preserve index usage:

```sql
CREATE INDEX idx_orders_year ON orders(EXTRACT(YEAR FROM order_date));
```

Re-run the `EXTRACT` version and verify whether the planner now uses this index. Note any difference compared to the range predicate approach.

## Task 4: Subquery vs. JOIN Performance

### Objective
Compare the execution strategies of an `IN` subquery and an equivalent `JOIN` when retrieving users who placed high-value orders.

---

### Step 1: Run the Subquery Version

```sql
EXPLAIN ANALYZE

FROM users
```

## Task 5: Composite Index Design

### Objective
Design a composite index that satisfies a multi-column filter and sort operation in a single index scan, and observe the elimination of an explicit sort step.

---

### Step 1: Run the Baseline Query

```sql
EXPLAIN ANALYZE

FROM orders
WHERE user_id = 'usr_0100'
  AND order_status = 'completed'
ORDER BY order_date DESC;
```

Record whether a `Sort` node appears in the plan and the total runtime.

---

### Step 2: Create a Basic Composite Index

```sql

```

Re-run the query. Check whether the `Sort` step has been eliminated.

---

### Step 3: Extend the Index to Cover the Sort

